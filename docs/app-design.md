# VoiceSync アプリ設計（音声認識・表示・事前学習）

ログインしたユーザーが実際に運用する本体アプリ（PCでマイク音声を取り込み、音声認識で文字化し、HDMI外部モニタに表示するアプリ）の設計メモ。マーケティングサイト（index.html 等）とは別の、実行環境側の設計。

## 1. 前提・要件

- PC（Windows/Mac想定）にマイクを接続し、音声をリアルタイムでテキスト化
- 外部ディスプレイにHDMI接続し、文字を表示
- 表示したテキストはログとして残す
- 事前に台本・資料（PDF/PPT/Word）を読み込ませ、専門用語や人名の認識精度を上げる「事前学習」機能を持つ
- 表彰式などでは人名を一字一句間違えずに表示する必要がある
- 将来的にオフライン（ネット接続なし）でも動作させたい

## 2. システムアーキテクチャ（3案の比較）

| 案 | 音声認識エンジン | オフライン対応 | 精度 | 備考 |
|---|---|---|---|---|
| ① クラウドSTT型 | Google Speech-to-Text / Azure Speech / Whisper API 等にストリーミング送信 | 不可（要ネット） | 高い | 開発は速いが、後からオフライン化するにはエンジンごと作り直しが必要 |
| ② ローカル完結型 | Whisper.cpp / Vosk / OS標準音声認識をPC上で実行 | 最初から可能 | PCスペック依存 | 通信費・API費用ゼロ、個人情報が外部に出ない |
| ③ ハイブリッド型（推奨） | 認識エンジンを抽象化し、クラウド／ローカルを設定で切替可能にする | 段階的に対応可能 | 用途に応じて選択 | 最初はクラウドで高精度リリースし、後からローカルエンジンを追加実装できる |

**採用方針: ③ ハイブリッド型。**
`SpeechRecognizer` のような共通インターフェースを定義し、`CloudEngine` と `LocalEngine`（Whisper.cpp / Vosk）を実装として差し替え可能にする。事前学習で作る「優先辞書」も、この抽象化層の上に共通で乗せる。

表示（HDMI出力）は方式に関係なく共通：OSの拡張ディスプレイ機能を使い、外部モニタ上に表示用ウィンドウをフルスクリーン表示する。

## 3. オフラインAIの実現可否

事前学習パイプラインの各工程は、すべてオフラインの技術だけで構成可能。

| 工程 | オフライン手段 |
|---|---|
| 資料の読み込み（PDF/PPT/Word） | `pdfplumber` / `python-docx` / `python-pptx` 等でローカル抽出 |
| 関連ワード抽出 | 日本語形態素解析（`MeCab`/`Sudachi`/`GiNZA`）＋ `TF-IDF` や `YAKE`。高精度化には小型ローカルLLM（`Phi-3-mini`, `Qwen2.5` 3B前後）も選択肢 |
| 個人情報の自動マスキング | `Presidio`（ルールベース＋NER）をローカル実行。日本語は氏名・電話番号等の検出精度向上のため辞書・正規表現を追加調整 |
| 音声認識時の文脈活用 | `Vosk` の grammar 機能 ＋ 読みベースの後処理補正（登録済み用語との照合・置換）。詳細は下記「PoC検証で判明した修正点」を参照 |
| ログ保存 | `SQLite` 等にローカル保存 |

制約は技術面ではなくPCスペック（特にGPUの有無）。リアルタイム性が必須の講演会用途では、軽量エンジン（Vosk）中心の設計と、高精度だが重いエンジン（Whisper＋LLM補正）の設計を、ハードウェアに応じて切り替えられるようにしておく。

### PoC検証で判明した修正点（重要）

`poc/vosk-check/` でVoskの実機検証を行った結果、当初の想定を修正する必要があることが分かった。

- **Voskのgrammar機能（辞書バイアス）は「モデルが元々知っている単語の中から絞り込む」仕組みであり、モデルの語彙に存在しない単語（外字を含む珍しい人名など）を新規に登録することはできない**（`Ignoring word missing in vocabulary` として無視される）。同音異義語の2択を絞り込むような用途には有効だが、「事前学習資料から抽出した任意の人名をVoskに教え込む」という当初の想定は成立しない
- そのため、**人名・専門用語の正確な表示は、Voskの生の認識結果に対して読み（ふりがな）ベースで照合し、登録済み表記に置換する後処理でしか実現できない**と考えたほうがよい（`pykakasi` 等で読みを比較する簡易プロトタイプで単一語の補正は成功を確認済み。複数語が近接するケースの実装は要改善）
- 4章・5章の「表示許可リスト」の運用は、上記を踏まえ「Voskへの辞書投入」ではなく「後処理補正の照合辞書」として位置づける

## 4. 個人情報保護と「表示許可リスト」の切り分け

「個人情報だから全部隠す」ではなく、**主催者が明示的に許可した固有名詞は正確に表示し、それ以外の個人情報は自動保護する**という2階建て設計。

| 種類 | 用途 | 扱い |
|---|---|---|
| 表示許可リスト | 受賞者名・登壇者名など、正確に表示したいもの | 主催者が事前に選択・登録。匿名化の対象外。音声認識の最優先辞書として利用 |
| 一般事前学習資料 | 台本・専門用語学習用のPDF/PPT/Word全般 | 個人情報が偶然含まれていても自動マスキング。表示・ログに残さない |

ログについても同様に、表示許可リストに載っている名前はログに残し、自動マスキング対象はログにも出さない。

## 5. 名前登録フロー（表示許可リストの作り方）

資料から名前候補を自動抽出し、主催者がチェックして選ぶ方式を採用。手入力は行わず、抽出結果からの選択を基本とする（ただし後述の例外あり）。

```
資料アップロード（PDF/PPT/Word）
  ↓
各ページ／スライドごとにテキスト層の有無を判定
  │
  ├─ テキスト層あり
  │    ├─ 表構造がある → 「氏名」「ふりがな」列を優先抽出（表彰リスト等はこの形式が多い）
  │    └─ 自由記述のみ → 日本語NER（GiNZA/ja_ginza）で人名候補を検出・重複排除
  │
  └─ テキスト層なし（スキャン画像・埋め込み画像等）
       → 「未読み込み画像」キューに保存（元のページ／スライド番号を記録）
  ↓
候補一覧を主催者に提示 → チェックボックスで表示対象を選択
  ↓
読み仮名の確認・修正
  ・資料にルビ情報があれば優先採用
  ・なければ pykakasi 等で自動推定し、候補として提示
  ・人名の読みは推定が外れやすいため、主催者による確認・修正を必須とする
  ↓
表示許可リストに登録
```

### スコープの決定（今回の初期実装範囲）

- OCRは実装しない。テキストとして抽出できる範囲のみを自動処理対象とする
- スキャン画像・埋め込み画像などテキスト層がない部分は自動処理せず、手動入力の対象として扱う

## 6. 手動入力フロー（自動抽出できなかった名前）

自動抽出できなかった名前は、単なる空欄入力ではなく、**該当画像を見ながら書き写せる**UIにする。

```
[未読み込み画像一覧]（例: 3件 未処理）

┌─────────────┐
│  [画像サムネイル]  │  p.12（表彰状スキャン）
└─────────────┘
  氏名：[________]　読み：[________]  [登録して次へ]

┌─────────────┐
│  [画像サムネイル]  │  スライド8
└─────────────┘
  氏名：[________]　読み：[________]  [登録して次へ]
```

- サムネイルを選ぶと画像を大きく表示し、隣に入力フォームを配置
- 登録が完了した項目はキューから消え、自動抽出分と同じ「表示許可リスト」に合流
- 元のページ／スライド番号を保持し、判読しづらい場合は元資料を参照できるようにする

## 7. 外字（読み込めない文字）への対応

日本語氏名では異体字（外字）が頻出するため、次の2段構えで対応する。

1. **Unicodeには存在するがフォントに字形がない場合**
   表示・ログ用フォントに `IPAmj明朝`（人名用外字を広くカバー）を採用し、四角い豆腐表示を防ぐ
2. **抽出自体が失敗する場合（画像埋め込み・スキャン起因）**
   完全自動化は諦め、6章の手動入力フロー（画像を見ながら書き写す）で対応する。主催者のIMEを使えば外字入力も現実的に可能
3. **音声認識との連携**
   AIが音声から認識するのは標準的な読み・漢字になるため、認識結果と登録済み氏名の「読み」を照合し、表示時のみ登録済みの外字表記に置き換える後処理を行う
